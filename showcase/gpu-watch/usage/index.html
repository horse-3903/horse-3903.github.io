gpu-watch: Usage Guide

Installation, configuration, CLI flags, notification setup, and real-world usage examples.

7 June 2026

Installation

Requires Debian/Ubuntu, bash, and python3. nvidia-smi is optional (GPU telemetry is disabled automatically if not found).

git clone https://github.com/horse-3903/gpu-watch.git
cd gpu-watch
sudo make install

make install copies gpu_watch.sh to /usr/local/bin/gpu-watch and gpu_watch_metrics.py to /usr/local/lib/gpu-watch/.

Basic usage

gpu-watch [OPTIONS] -- COMMAND [ARGS...]

The -- separator is required to distinguish gpu-watch flags from the wrapped command’s arguments.

CLI flags

FlagDefaultDescription
--topic TOPIC:ntfy.sh topic for push notifications. Omit to disable notifications.
--name NAMEderived from commandHuman-readable run name used in log directory and notification titles.
--log-dir DIR./logsBase directory for run logs.
--interval N300Notification interval in seconds (how often status pings are sent).
--gpu-interval N30nvidia-smi polling interval in seconds.
--no-gpu:Disable GPU telemetry entirely (useful on CPU-only machines).
--top-k N8Maximum number of metrics to include in notifications.
--ntfy-url URLhttps://ntfy.shSelf-hosted ntfy server URL.

Notification setup

gpu-watch uses ntfy.sh for push notifications. No account needed: just pick a unique topic name and subscribe to it on your phone or browser.

# Install the ntfy app, subscribe to "my-training-topic", then:
gpu-watch --topic my-training-topic -- python train.py

You’ll receive notifications for:

  • STARTED: command, host, log path
  • STATUS (every --interval seconds): elapsed time, current metrics
  • SUCCESS: total duration, exit code 0, final metrics
  • FAILURE: total duration, exit code, tail of stderr

For a self-hosted ntfy instance:

gpu-watch --topic my-topic --ntfy-url https://ntfy.example.com -- python train.py

Multi-phase training

Pass multiple commands separated by -- to run them sequentially. Each phase is notified and logged independently:

gpu-watch --topic my-topic \
  -- python pretrain.py --epochs 10 \
  -- python finetune.py --epochs 5

If any phase fails, subsequent phases are skipped and a failure notification is sent.

Metric extraction

gpu_watch_metrics.py tries four sources in order and returns the first that yields results:

  1. TensorBoard event files: scans for events.out.tfevents.* in the log directory
  2. CSV files: Ultralytics results.csv and HuggingFace trainer state JSON
  3. JSON/JSONL files: any *.json or *.jsonl in the run directory
  4. stdout regex: regex over the last 500 lines of train.log

The stdout regex looks for key=value or key: value patterns and ranks results by a priority list:

epoch → step → val/loss → val/accuracy → mAP50-95 → mAP50 →
f1 → accuracy → loss → lr → ...

Metrics matching skip patterns (timestamps, throughput, speed) are excluded.

Real-world examples

Fine-tuning a YOLOv8 model on RunPod:

gpu-watch --topic runpod-yolo --name yolov8-finetune \
  -- yolo train model=yolov8n.pt data=custom.yaml epochs=100 imgsz=640

Multi-GPU distributed training:

gpu-watch --topic dist-train --gpu-interval 10 \
  -- torchrun --nproc_per_node=4 train_ddp.py --config cfg.yaml

CPU-only debugging run:

gpu-watch --topic debug --no-gpu --interval 60 \
  -- python train.py --debug --epochs 2

Log directory structure

logs/
└── yolov8-finetune_20260607_143210/
    ├── train.log           ← complete stdout + stderr
    ├── gpu.log             ← nvidia-smi snapshots (one per --gpu-interval)
    ├── metrics.log         ← extracted metrics at each notification cycle
    ├── summary.txt         ← name, command, host, start/end time, duration, exit code
    ├── exit_code.txt       ← just the integer exit code
    ├── command.txt         ← exact command string
    ├── environment.txt     ← hostname, CUDA_VISIBLE_DEVICES, K8s pod name, etc.
    ├── pip_freeze.txt      ← pip freeze output at job start
    ├── nvidia_smi_start.txt
    └── nvidia_smi_end.txt

The log directory is created before the job starts and is safe to inspect while the job is running.
