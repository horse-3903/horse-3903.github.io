gpu-watch: Overview

Project overview: what gpu-watch does, how it works, and the problems it solves.

7 June 2026

What is gpu-watch?

gpu-watch is a lightweight CLI wrapper for long-running GPU training jobs. You prefix any training command with gpu-watch and it handles logging, telemetry, metric extraction, and push notifications: with no changes to your training code.

gpu-watch --topic my-topic -- python train.py --epochs 100

The core problem it solves: GPU training jobs are fragile and slow. You start a run, walk away, and have no idea whether it’s progressing, hung, or crashed: until you SSH back in and scroll through a terminal. gpu-watch makes training jobs observable without requiring you to instrument your code.

Architecture

gpu-watch
├── gpu_watch.sh           ← main wrapper script
├── gpu_watch_metrics.py   ← multi-source metric extractor
├── Makefile               ← install target (copies to /usr/local/bin)
└── logs/
    └── <name>_<timestamp>/
        ├── train.log           ← full stdout + stderr (via FIFO + tee)
        ├── gpu.log             ← periodic nvidia-smi snapshots
        ├── metrics.log         ← extracted metrics per notification cycle
        ├── summary.txt         ← run metadata (host, duration, exit code)
        ├── exit_code.txt
        ├── command.txt
        ├── environment.txt     ← hostname, CUDA_VISIBLE_DEVICES, etc.
        ├── pip_freeze.txt
        ├── nvidia_smi_start.txt
        └── nvidia_smi_end.txt

Every run gets its own timestamped directory under logs/. Logs persist after the terminal session ends, including across SSH disconnects.

Key features

Persistent timestamped run directories: every invocation creates logs/<name>_<timestamp>/, so you never lose a run’s output even if the terminal closes.

Live log streaming via FIFO + tee: gpu-watch creates a named pipe and uses tee to write stdout/stderr to both the terminal and the log file in real time.

GPU telemetry via nvidia-smi: a background polling loop snapshots nvidia-smi every N seconds into gpu.log, capturing memory usage, temperature, utilisation, and power draw. Full snapshots at job start and end are saved separately.

Multi-source metric extraction: gpu_watch_metrics.py extracts metrics from four sources in priority order: TensorBoard event files, CSV files (Ultralytics/HuggingFace format), JSON/JSONL files, and stdout regex. Results are ranked by a priority list (epoch → validation metrics → mAP → loss → lr) and truncated to the top-k keys.

ntfy.sh push notifications: sends notifications on job start, periodic status (with current metrics), success, and failure. Works with any ntfy.sh topic; no account required.

Exit code propagation: the wrapped process’s exit code is preserved and written to exit_code.txt. gpu-watch exits with the same code, so it’s safe to use in scripts.

Signal handling: SIGINT and SIGTERM are forwarded to the wrapped process and the background polling loop is cleaned up gracefully.

Multi-phase sequential training: pass multiple -- separated commands to run them in sequence; each phase is tracked and notified independently.

Kubernetes-aware: reads KUBERNETES_POD_NAME and related env vars to include pod context in notifications and logs.

When to use it

  • Long fine-tuning or pre-training runs on remote GPUs (RunPod, Lambda, cloud VMs)
  • Any job where you want to walk away and be notified when it finishes
  • Debugging training instability: the persistent logs capture the exact output at the time of a crash
  • Multi-experiment pipelines where you want per-run telemetry without setting up a full MLOps stack
